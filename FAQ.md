# SmartReminder — Known Issues

Список известных проблем. Актуален для версии бота и CLI по состоянию на июль 2025.

---

## ISSUE-001: Telegram-бот не отвечает после перезапуска процесса

**Компонент:** Telegram-бот  
**Платформа:** все ОС  
**Статус:** известная проблема, в работе

**Симптом:**  
После остановки и повторного запуска бота (`Ctrl+C` + перезапуск) бот принимает сообщения, но не отвечает. В логах нет ошибок, `[TelegramBot] Listening for messages...` отображается.

**Шаги воспроизведения:**
1. Запустить бота.
2. Отправить сообщение — ответ приходит нормально.
3. Остановить процесс (`Ctrl+C`).
4. Запустить снова.
5. Отправить сообщение — ответа нет.

**Причина:** При остановке Telegram не успевает сдвинуть offset; после рестарта бот получает старые апдейты и застревает в обработке очереди.

**Обходное решение:** Подождать ~30 секунд после рестарта или перезапустить ещё раз.

## ISSUE-002: Assist-режим зависает и не возвращает ответ

**Компонент:** ToolCallingAgent, ToolCallingLoop  
**Платформа:** все ОС  
**Статус:** известная проблема

**Симптом:**  
При использовании assist-режима агент начинает обработку, но ответ не приходит. Пользователь видит статус "в очереди" бесконечно долго.

**Шаги воспроизведения:**
1. Отправить сложный запрос, требующий нескольких вызовов инструментов (например, "найди все файлы с ошибками и создай тикет по каждой").
2. Агент вызывает инструменты итеративно.
3. На 4-й или 5-й итерации перестаёт отвечать.

**Причина:** LLM возвращает ответ в неожиданном формате (не TOOL_CALL и не FINAL_ANSWER) на граничных итерациях, `parseErrorCount` исчерпывает лимит, но fallback message не отправляется корректно.

**Обходное решение:** Переформулировать запрос более конкретно. Разбить на несколько шагов.

## ISSUE-003: RAG возвращает пустой контекст при первом запросе после запуска

**Компонент:** ProjectKnowledgeService, RagSearcher  
**Платформа:** все ОС  
**Статус:** известная проблема

**Симптом:**  
После запуска бота и выполнения `/init` первый вопрос, требующий RAG, возвращает ответ без контекста документации. Последующие вопросы работают нормально.

**Шаги воспроизведения:**
1. Запустить бота.
2. Выполнить `/init owner/repo`.
3. Сразу задать вопрос по коду репозитория.
4. Агент отвечает без использования документации.
5. Задать тот же вопрос повторно — ответ содержит контекст из RAG.

**Причина:** Индекс загружается асинхронно; первый `getContext()` вызывается до завершения загрузки эмбеддингов в память.

**Обходное решение:** Подождать 5–10 секунд после `/init` перед первым вопросом.

## ISSUE-004: Бот не запускается на Windows 10/11 через CMD

**Компонент:** Запуск, gradlew  
**Платформа:** Windows 10, Windows 11  
**Статус:** известная проблема

**Симптом:**  
Попытка запустить бота через `gradlew :telegram:run` в CMD завершается ошибкой `'TELEGRAM_BOT_TOKEN' is not recognized as an internal or external command`.

**Шаги воспроизведения:**
1. Windows 10 или 11, CMD (не PowerShell).
2. Выполнить: `TELEGRAM_BOT_TOKEN=xxx gradlew :telegram:run`
3. Ошибка: `'TELEGRAM_BOT_TOKEN' is not recognized...`

**Причина:** CMD не поддерживает синтаксис `KEY=VALUE command`. Работает только в PowerShell или Git Bash.

**Обходное решение:**  
PowerShell: `$env:TELEGRAM_BOT_TOKEN="xxx"; .\gradlew :telegram:run`  
Git Bash: `TELEGRAM_BOT_TOKEN=xxx ./gradlew :telegram:run`

## ISSUE-005: GitHub MCP не подключается на macOS Apple Silicon (M1/M2/M3)

**Компонент:** McpManager, GitHub MCP сервер  
**Платформа:** macOS 13+, Apple Silicon  
**Статус:** известная проблема, временное решение есть

**Симптом:**  
При наличии `GITHUB_PERSONAL_ACCESS_TOKEN` бот пытается подключить GitHub MCP, но подключение падает с ошибкой в логах:

```
[MCP] Failed to connect to github: initialize timed out — is the server running?
[stderr] Error: Cannot find module ...
```

**Шаги воспроизведения:**
1. Mac с чипом Apple Silicon (M1/M2/M3).
2. Добавить `GITHUB_PERSONAL_ACCESS_TOKEN` в `local.properties`.
3. Запустить бота.
4. В логах — ошибка подключения GitHub MCP.

**Причина:** `npx` на Apple Silicon иногда скачивает x86 версию `@modelcontextprotocol/server-github`, которая не запускается без Rosetta 2.

**Обходное решение:** Установить Rosetta 2: `softwareupdate --install-rosetta` или установить Node.js native arm64 через `nvm`.

## ISSUE-006: История чата сбрасывается при потере соединения с Telegram

**Компонент:** TelegramBotRunner, ToolCallingAgent (ChatHistory)  
**Платформа:** все ОС  
**Статус:** известная проблема, не планируется к исправлению в текущей версии

**Симптом:**  
Если соединение с Telegram API прерывается (сеть упала, timeout) и бот продолжает работу, история переписки для всех пользователей обнуляется.

**Шаги воспроизведения:**
1. Провести несколько обменов сообщениями с ботом.
2. Временно отключить интернет на сервере (или дождаться timeout).
3. Восстановить соединение.
4. Отправить новое сообщение — бот не помнит предыдущий контекст.

**Причина:** `ChatHistory` хранится в памяти (`ConcurrentHashMap` в `ToolCallingAgent`). При исключении в polling loop `getUpdates()` возвращает пустой список, но при некоторых сетевых ошибках процесс перезапускает coroutine scope, сбрасывая состояние.

**Обходное решение:** История не персистируется между сессиями — это известное ограничение текущей версии.
